name: tidepo-step-plan-refactor
description: TiDepo repo専用。実装方針・手順書から、指定された1手順の詳細実装計画書を、周辺コードのリファクタ候補探索込みで作成する。実装/バグ修正と周辺リファクタを同一手順で同時に行う前提の計画にする。Plan Modeと併用する場合は実装・ファイル編集はしない（探索のための読み取りのみ行う）。
---

## 入力

- 実装方針・手順書のパス（`docs/plan/` 配下）
- 対象手順番号
- 関連設計書のパス（`docs/design/` 配下）
- 関連Issue番号がある場合はその番号
- リファクタで特に注目したい範囲・観点がある場合はその指定（任意）

## 基本方針

- 「実装/バグ修正」と「周辺リファクタ」を同一手順で同時に行う前提の詳細実装計画書の作成を主目的にする。出力先は既定で `docs/plan/`。
- Plan Modeで呼ばれた場合は、実装やファイル編集をしない。周辺コードの読み取り（探索・分析）は行う。
- Issue更新などの外部副作用は既定では行わない。

## 探索方針

- 先に設計書、実装方針・手順書、既存実装/設定を読む。
- 対象手順が触れるコードだけでなく、その周辺（呼び出し元・呼び出し先、同一モジュール、同種パターンの実装）を実際に読み、リファクタ候補を洗い出す。
- repo/systemから分かることは質問せず探索で解消する。
- 高影響の未決事項は必ず質問する（スコープ、成功条件、副作用許容、出力先、検証方法、リファクタ範囲の許容）。
- 不明点があれば質問して解消し、未決事項をなくす。

## リファクタ候補の観点

- 重複コード・近接した同種ロジックの集約余地。
- 命名・責務の不一致、レイヤー違反（Backend オニオンの依存方向 / Frontend FSD の依存方向・public API 境界）。
- 複雑度過多、死蔵コード・未使用要素・過剰な分岐。
- 型・例外・コメントの規約逸脱、テスト不足・テスト容易性を阻害する構造。
- 各候補を「変更に伴い必須」「任意改善（今回やる）」「将来TODO（今回対象外）」に区分する。

## リファクタ範囲の原則

- 対象手順の実装/修正と密接に関係する範囲に限定する。無関係な大規模リファクタは別手順に切る。
- リファクタはレビュー可能な粒度に保つ。
- 振る舞いを変えないリファクタと、振る舞いを変える実装/修正を計画上明確に区別する（コミット境界の指針も書く）。

## 手順

1. 実装方針・手順書と対象手順を読む。
2. 関連設計書、既存docs、既存実装/設定を必要範囲で読む。
3. 対象手順が触れるコードと、その周辺コードを実際に読み、リファクタ候補を洗い出す。
4. 対象手順の目的、完了条件、対象外、既知制約を整理する。
5. リファクタ候補を「必須/任意/将来TODO」に区分し、今回の対象範囲を確定する。
6. 変更予定を Backend（オニオン各層）/ Frontend（FSDスライス）/ Test / Docs/Ops に、実装分とリファクタ分で分けて整理する。
7. 変更予定ファイルを列挙する（実装起因 / リファクタ起因のラベル付き）。
8. 高影響の未決事項がある場合は質問する。
9. テスト方針と品質ゲートを明記する（リファクタが振る舞いを変えないことを既存テストで担保する点を含む）。
10. 詳細実装計画書を作成または更新する。

## 詳細実装計画書の出力項目

- 対象Issue / 対象手順 / 目的 / 完了条件
- リファクタ候補と区分（必須/任意/将来TODO・各々の理由とリスク）
- 今回のリファクタ対象範囲
- 変更予定ファイル（実装起因 / リファクタ起因）
- レイヤー別実装方針（実装 + リファクタ）
- 実装とリファクタの実行順序・境界（コミット指針）
- 実行方針 / テスト方針（振る舞い不変の担保を含む）/ 品質ゲート
- 未決事項 / 実装開始前確認

## 品質ゲート基準（プロジェクトのbuild設定に従う）

- Backend（Gradle）: `./gradlew spotlessApply` → `./gradlew check` → ArchUnitオニオン依存検証 → `./gradlew test`。
- Frontend: `npm run lint` → `npm run typecheck` → `npm test`（Vitest）→ `npm run build`。

## Plan Modeで呼ばれた場合

- 実装やファイル編集はしない（周辺コードの読み取り＝探索・分析のみ行う）。
- 詳細実装計画書の作成計画までに留める。
- 公式の最終計画は ExitPlanMode で提示する。

## 出力

- 作成/更新した詳細実装計画書パス
- 洗い出したリファクタ候補と今回の対象範囲
- 未決事項
- 次に実行する場合の注意（通常は `tidepo-step-dev`）
