PR番号を受け取り、以下の3フェーズで「行単位のインラインレビュー」を行う。リポジトリは `ogihaya/TiDepo`。

1. **徹底洗い出し**: 問題点を網羅的に抽出する（粗くてよい・件数を稼ぐ）
2. **ダブルチェック**: 各指摘が本当に正しいかを実コードで再検証し、誤検知を落とす
3. **インラインコメント投稿**: 確証が取れた指摘だけを、該当ファイルの該当行に紐付けて投稿する

`tidepo-review` との違いは、**最終成果物が「コード行に紐づくインラインコメント」である点**と、**投稿前に全指摘を必ず検証（ダブルチェック）する点**。

## フェーズ0: 情報収集

```
gh pr view {番号} --repo ogihaya/TiDepo
gh pr view {番号} --repo ogihaya/TiDepo --json headRefOid,baseRefName,files
gh pr diff {番号} --repo ogihaya/TiDepo
```

- PRのタイトル・説明・関連Issueを把握する。
- 変更ファイル一覧を取得する。
- **`headRefOid`（head SHA）を控える** — インラインコメント投稿に必須。
- 差分が大きく出力が切れる場合はファイルへ保存し、Readでページングして全文を読む。
- 関連設計書が `docs/design/` / `docs/plan/`、要件が `docs/requirements.md` にあれば読み、要件との齟齬も検出対象にする。

> **重要（インラインコメントの制約）**: GitHub のインラインコメントは **diff のハンク内に現れる行**（追加行・変更行・表示されている文脈行）にしか付けられない。差分に出ていない行へのコメントは 422 で失敗する。フェーズ0で各ファイルの diff ハンクを把握し、コメント可能な行番号（新ファイル側 = RIGHT）の範囲を意識しておくこと。

## フェーズ1: 徹底洗い出し（網羅性を優先）

このフェーズは**件数を稼ぐ**フェーズ。確信度は問わず候補として列挙する。落とすのは次フェーズ。

変更規模が大きい場合は `Agent`（Explore / `tidepo-code-explorer` など）を**複数並列**で起動し、観点ごとに分担する。例: 「正しさ/バグ」「セキュリティ」「パフォーマンス」「影響範囲・破壊的変更」「オニオン/FSD逸脱」「設計書との齟齬」。

抽出観点:

- **正しさ・バグ**: ロジック誤り、境界条件、null/空、例外時挙動、非同期/トランザクション/排他。猶予時間計算・繰り返し予定の発生判定は特に注意。
- **アーキテクチャ**: Backend オニオン依存方向違反・レイヤー責務混在、Frontend FSD 依存方向違反・public API 越境。
- **品質**: 既存スタイル不整合、命名、不要コード・デバッグ残骸。
- **セキュリティ**: SQLインジェクション・XSS、認証/認可漏れ、機密情報のハードコード、入力未検証。
- **パフォーマンス**: N+1、不要な全件取得、索引のない検索条件。
- **影響範囲**: 既存APIの破壊的変更、共有スライス/共通モジュールへの波及、DBスキーマ変更の既存データ影響、マイグレーション整合。

各候補は次の形式でメモする:
```
[候補] <一言要約>
  ファイル: <path>:<行番号(新ファイル側)>
  観点: 正しさ / アーキ / 品質 / セキュリティ / 性能 / 影響範囲 / 設計齟齬
  根拠(仮): <なぜ問題だと思うか>
```

## フェーズ2: ダブルチェック（誤検知の排除・最重要）

**フェーズ1の各候補を、投稿せずに必ず一件ずつ検証する。** ここを省略しない。誤った指摘はレビュー全体の信頼を損なう。

各候補に対して実コードで確認する:

1. 実ファイルを Read で開いて文脈を確認（diffの断片だけで判断しない）。
2. 呼び出し元・呼び出し先を Grep で追う（「未使用」「破壊的変更」系は消費箇所を実際に検索）。
3. 既存実装・既存パターンと照合（「規約違反」系は同種コードを確認）。
4. 型・スキーマ・マイグレーションの実体を確認（git履歴 `git show main:<path>`, `git log -- <path>` で裏取り）。
5. 行番号が diff ハンク内（コメント可能な行）か確認。範囲外なら最も近い差分行に寄せるか、サマリ送りにする。

各候補を分類する:

- **確証あり（投稿する）**: 重大度 `[必須] / [推奨] / [提案]` を付与。
- **誤検知（破棄）**: 破棄理由を内部メモに残す。
- **インライン不可（サマリへ）**: 設計レベル・ファイル横断・デプロイ注意などはレビュー本文へ。

> ダブルチェックを厚くしたい場合は、各「確証あり」候補について `Agent` で**反証役**を1体立て、「この指摘は本当に問題か？反証してみよ」と問わせ、覆らなかったものだけを残す。

検証後、確定した指摘を一覧化して**投稿前サマリ**を提示する:
```
## インラインレビュー候補（投稿前確認）
| # | 重大度 | ファイル:行 | 指摘 | 検証で確認した事実 |
|---|--------|------------|------|-------------------|
（破棄した候補とその理由も末尾に簡潔に列挙）
```

## フェーズ3: インラインコメント投稿

確証が取れた指摘を、GitHub Review API で**1回のレビューにまとめて**該当行へ投稿する。

1. head SHA を取得（未取得なら）:
   ```
   COMMIT_ID=$(gh pr view {番号} --repo ogihaya/TiDepo --json headRefOid -q .headRefOid)
   ```

2. レビューペイロードJSONを `Write` で作成する（例: scratchpad に `inline-review-{番号}.json`）:
   ```json
   {
     "commit_id": "<COMMIT_ID>",
     "body": "## インラインレビュー結果\n\n（全体所感・良い点・インライン不可の指摘をここに書く）",
     "event": "COMMENT",
     "comments": [
       { "path": "backend/.../Foo.java", "line": 42, "side": "RIGHT", "body": "**[必須]** ..." },
       { "path": "frontend/src/.../bar.ts", "start_line": 10, "line": 14, "side": "RIGHT", "body": "**[提案]** ..." }
     ]
   }
   ```
   - `line`: 新ファイル側の行番号。`side` 省略時は `RIGHT`。
   - 削除行に付けたい場合のみ `"side": "LEFT"`。
   - 複数行に跨ぐ場合は `start_line` + `line`。
   - `event` は `COMMENT` / `REQUEST_CHANGES` / `APPROVE` から選ぶ。

3. 投稿（実行直前にユーザー確認）:
   ```
   gh api repos/ogihaya/TiDepo/pulls/{番号}/reviews --method POST --input <JSONパス>
   ```

4. 422 エラーはほぼ「その行が diff ハンク外」が原因。`line` を差分内へ寄せるか、本文（`body`）側へ移して再投稿する。

## インラインコメントの書き方

- 先頭に重大度: `**[必須]**` / `**[推奨]**` / `**[提案]**`。
- **何が問題か → なぜ問題か → どう直すか** を簡潔に。必要なら修正コード例をコードブロックで添える。
- 設計書と矛盾する場合は該当箇所を引用する。
- 質問形でなく、フェーズ2で確認した事実を根拠に断定する。

## 注意

- **フェーズ2（ダブルチェック）を絶対に飛ばさない。** 未検証の指摘を投稿しない。
- 誤検知ゼロを優先する。確信が持てない指摘は「提案」に格下げするか本文へ。
- 些細なスタイルだけの指摘は控え、ロジック・設計・安全性・影響範囲を優先する。
- 投稿は原則1レビューにまとめる。
- 投稿は GitHub 状態を変える操作なので、実行直前にユーザー確認する。
- 全ての出力・コメントは日本語で行う。
