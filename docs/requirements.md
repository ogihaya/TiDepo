# TiDepo 要件定義書

最終更新: 2026-06-27
ステータス: ドラフト（改訂版 v4）

> v2 改訂概要: 初版（参考リポジトリ相当のMVP）に対し、ユーザーレビューコメントを反映してスコープを拡張した。主な変更は、認証/ロール管理（OAuth2.0/PKCE/Spring Security）、タスク入れ子分割、タスク完了、Googleカレンダーの双方向・自動同期、Google Tasks との双方向同期、RDB（PostgreSQL想定）への刷新、変更履歴の監査、レスポンシブWeb対応をMVPに取り込んだこと。
>
> v3 改訂概要: プロジェクト概念を廃止し、タスクの入れ子（親子）ツリーへ一本化（最上位タスクがプロジェクトの役割を兼ねる）。ロールは ADMIN / USER のみとする。繰り返しは複雑な RRULE まで対応する方針とした。
>
> v4 改訂概要: アーキテクチャ方針を確定。優先軸は「学習の深さ」。認証はブラウザ↔サーバ間を BFF パターン（トークンはサーバ側保持、httpOnly セッションCookie。PKCE はサーバ↔Google）。IaC は AWS CDK（TypeScript）。ホスティングは AWS（ECS/Fargate + RDS）。Google 同期は腐敗防止層（ACL）として隔離。Calendar の自動同期は push(webhook) ではなく**ポーリング（syncToken 増分）に統一**（Tasks と方式を揃え、公開 webhook・channel 更新の運用を避ける）。バックエンドは Java を継続。

## 1. はじめに

### 1.1 本書の目的

本書は、ToDoリストカレンダーアプリ **TiDepo** の要件を定義する。プロダクトの目的、対象範囲、機能要件、非機能要件、技術・アーキテクチャ方針、外部連携、データモデル、コア計算ロジック（猶予時間）、制約、将来構想を整理し、設計・実装フェーズの起点とすることを目的とする。

本書は要件レベルの記述にとどめ、詳細設計（物理スキーマ、API設計、画面詳細、role権限マトリクス、同期アルゴリズム詳細等）は後続の設計フェーズで決定する。

### 1.2 背景

- 本プロジェクトは、趣味・個人開発として行うToDoリストカレンダーアプリである。学習目的（オニオンアーキテクチャ、FSD、OAuth2.0/PKCE、Spring Security、監査機能、Google連携、AWS/IaC など）も兼ねる。
- 同等のプロダクトを Next.js / React / TypeScript / Firebase 構成で一度実装済み（参考リポジトリ `todolistcalendar2`）であり、コア価値（猶予時間の算出・可視化）とコアロジックはそこで検証済みである。
- 今回はアーキテクチャと技術スタックを見直してゼロから作り直す。バックエンドを Java / Spring Boot、フロントエンドを TypeScript / React で再構築し、データストアは RDB（PostgreSQL想定）へ刷新する。
- 参考リポジトリのMVPに加え、学習目的を踏まえて認証/ロール管理・タスク入れ子分割・タスク完了・双方向自動同期・監査・レスポンシブ対応を今回のスコープへ拡張する。参考リポジトリの画面・データモデル・計算式は「検証済みの参考」であり、確定仕様ではない（より良い方式があれば採用する）。

### 1.3 プロダクトの一言要約

TiDepo は、予定とタスクを同じカレンダー上で管理し、締切までに **「本当に残っている作業可能時間（猶予時間）」** を可視化する個人向けタスク管理アプリである。単に締切日や予定を並べるのではなく、日々の予定で使えない時間・タスクの見積もり時間・予定が未確定な期間の見込み時間を踏まえて、タスクごとの猶予時間を表示する。

これにより、ユーザーは「どのタスクに早く着手すべきか」「今の予定量で締切に間に合うか」を判断できる。

## 2. 用語定義

| 用語 | 定義 |
| --- | --- |
| 予定（スケジュール / Schedule） | 開始・終了時刻が決まっている時間ブロック（会議、授業、外出など）。Google Calendar のイベントに対応する。**睡眠などの生活時間は予定として登録せず、「1日に使える時間」設定で吸収する**（睡眠が長い人は1日に使える時間を短く設定する、という考え方）。 |
| タスク（Task） | 締切日と所要時間（見積もり）を持つ作業項目。締切日にカレンダー表示され、猶予時間計算の対象となる。完了状態を持つ。親子（入れ子）構造を持てる。 |
| 親タスク / 子タスク（入れ子） | 大きなタスクを小さなタスクへ分割した階層構造。実所要時間を持つのは末端（葉）タスクで、親タスクの所要時間は子タスクの合計に一致する。最上位タスク（親を持たないタスク）は、複数タスクをまとめる「プロジェクト」的な役割を兼ねる（独立したプロジェクト概念は持たない）。 |
| 残余タスク | 親タスクの所要時間に対し、明示的な子タスクの合計が不足する分を吸収するためにシステムが自動生成する子タスク（葉タスク）。「親＝子の和」を保つために用いる。 |
| 一時的なタスク（TempTask） | 締切・所要時間が未定の用事を素早く記録する軽量メモ。後から本タスクへ変換できる。Google Tasks から取り込んだ所要時間未定のタスクもここに入る。 |
| 猶予時間（残り時間 / Remaining Time） | あるタスクの締切までに、予定で使う時間・他タスクの所要時間・未確定期間の見込みを差し引いた後に残る、実際に作業へ使える時間。負値は時間不足を意味する。 |
| 1日に使える時間（availableTimePerDay） | 睡眠等の生活時間を除き、1日のうち作業に充てられる時間の上限（時間単位）。ユーザー設定値。 |
| 予定確定最終日（dateTakeIntoAccount） | この日までは予定が確定済みとして扱う最終日。これ以降は予定が未入力でも見込みで計算する。**未設定にすると「無限に確定済みとして扱うモード」**となり、未確定期間の見込み補正を行わない。ユーザー設定値。 |
| 予定未確定日の1日あたり使える時間（availableTimePerUnscheduledDay） | 予定確定最終日より先の未確定期間について、1日あたり作業に使えると見込む時間（時間単位）。ユーザー設定値。 |
| サボってもいい時間 | 現在のタスク状況から算出される、1日平均で作業しなくても締切に間に合う余裕時間（補助的な指標）。 |
| ロール（Role） | ユーザーに付与する権限区分（ADMIN / USER）。操作可能なAPI・閲覧可能な画面を出し分ける。 |
| Google Calendar 同期 | Google Calendar の予定とアプリの予定を双方向・自動で同期する機能。 |
| Google Tasks 同期 | Google Tasks のタスクとアプリのタスクを双方向で同期する機能（ポーリング方式）。 |

## 3. 解決したいユーザー課題

- 期限が遠いタスクを後回しにしてしまい、締切直前まで危機感を持ちにくい。
- カレンダー上の予定とToDoリストが分かれているため、実際に作業へ使える時間が見えにくい。
- タスクの所要時間を考慮したとき、締切までに余裕があるのか不足しているのかを直感的に判断しにくい。
- 思いついた小さな用事をすぐ記録したいが、締切や所要時間まで決まっていない段階では正式なタスクにしづらい。
- 普段使っている Google Calendar / Google Tasks の予定・タスクも、アプリの猶予時間計算に反映したい。
- 大きなタスクを分割し、まとまり単位で進行状況を把握したい。

---

## 4. 想定ユーザー

- 個人の予定とタスクをまとめて管理したいユーザー。
- 学習、仕事、制作、事務作業など、締切と見積もり時間があるタスクを複数抱えるユーザー。
- 予定の空き時間ではなく、締切までに実際に使える作業時間を確認したいユーザー。
- タスク管理を義務感だけでなく、「時間を貯金する」感覚で前向きに続けたいユーザー。

MVPでは、各ユーザーが自分のデータを管理する **マルチユーザー（データはユーザー単位で分離）** を前提とし、ロールによる操作・画面の出し分けを行う。複数ユーザー間での実データ共有・コラボレーションは将来構想とする（[12. 将来構想](#12-将来構想)）。

## 5. スコープ

### 5.1 本リリース（MVP）に含むもの

- **認証・アカウント**: Google ログイン ＋ メール/パスワード認証（OAuth2.0 / PKCE / Spring Security）。ユーザー単位のデータ分離。ADMIN / USER のロールによる操作・画面の出し分け（管理者画面を含む）。
- **カレンダー表示**: 月／週／日など複数の表示（具体的なUIは設計フェーズで詰める）。
- **予定管理**: 追加・編集・削除。繰り返し（Google Calendar 互換、複雑な RRULE まで対応）、複数日対応。
- **タスク管理**: 追加・編集・削除。完了状態の管理。締切日・所要時間・メモ。
- **タスクの入れ子（親子分割）**: 大きなタスクを子タスクへ分割し、まとまり単位で集計・進行状況を確認（最上位タスクがプロジェクトの役割を兼ねる）。
- **一時的なタスク**: 記録・削除・本タスクへの変換。
- **猶予時間の可視化**: 完了タスクの除外、未確定期間の見込み、無限確定モードに対応。
- **設定**: 1日に使える時間、予定確定最終日（未設定で無限確定モード）、予定未確定日の1日あたり使える時間。
- **Google Calendar 同期**: 予定の双方向・自動同期（ポーリングによる syncToken 増分同期）。
- **Google Tasks 同期**: タスクの双方向同期（ポーリング、所要時間未定のGoogle由来タスクは一時タスク扱い、葉タスクのみ反映）。
- **監査**: 主要エンティティの変更履歴（誰がいつ何を変えたか）。
- **レスポンシブWeb**: PC／スマートフォン両方で主要操作が可能。

### 5.2 本リリースに含まないもの（将来構想）

[12. 将来構想](#12-将来構想) に記載の機能（複数ユーザー間での実データ共有・チーム/組織管理、AI支援、Slack 等の他サービス連携、PWA/ネイティブアプリ）は本リリースの対象外とする。ただし、データモデル・アーキテクチャはこれらの拡張を阻害しない構造を目指す。

## 6. 機能要件

要件IDは `FR-<領域>-<連番>` 形式で付与する。非機能要件は [10.5](#105-非機能要件nfr) を参照。

### 6.1 認証・アカウント（FR-AUTH）

| ID | 要件 |
| --- | --- |
| FR-AUTH-01 | ユーザーは Google アカウント、またはメール/パスワードでサインアップ・ログイン・ログアウトできる。 |
| FR-AUTH-02 | 認証は OAuth2.0 / OIDC を基本とし、認可コードフロー＋PKCE を用いる。ブラウザ（React SPA）↔サーバ間は BFF パターンとし、OAuth トークンはサーバ側で保持して、ブラウザには httpOnly セッションCookie のみを渡す（トークンをブラウザに置かない）。PKCE はサーバ↔外部IdP/Google で用いる。 |
| FR-AUTH-03 | 未ログイン状態で保護対象画面へアクセスした場合は、ログイン画面へ誘導する。 |
| FR-AUTH-04 | 予定・タスク・一時タスク・設定の参照・保存操作は、ログイン済みユーザー自身のデータに対してのみ行える（ユーザー単位のデータ分離）。 |
| FR-AUTH-05 | Google Calendar / Google Tasks 連携には、ログイン手段とは別に Google への連携認可（該当スコープ）が必要となる。ユーザーは連携を許可・解除できる。 |

補足: ログイン手段が Google でない場合でも、Google 同期を使うには Google 連携認可が別途必要となる。

### 6.2 ロール・管理者（FR-ROLE）

ロールは ADMIN / USER の2種類とする。

| ID | 要件 |
| --- | --- |
| FR-ROLE-01 | ユーザーは ADMIN または USER のロールを持つ。 |
| FR-ROLE-02 | ロールに応じて、操作可能なAPI・閲覧可能な画面を出し分ける。 |
| FR-ROLE-03 | ADMIN は管理者画面で全ユーザーの一覧・基本情報・利用状況（統計）を閲覧できる。 |
| FR-ROLE-04 | USER は自分のデータのみ参照・編集でき、管理者画面にはアクセスできない。 |

補足: 現段階では実データのユーザー間共有が無いため、ロールは主に「管理者機能の出し分け」で意味を持つ。共有機能の実装時に権限差を拡張する。具体的な権限マトリクスは設計フェーズで確定する。

### 6.3 カレンダー表示（FR-CAL）

カレンダー表示は参考リポジトリと同一にはせず、使いやすさを優先して柔軟に設計する。Google Calendar のようなUIも候補とする。

| ID | 要件 |
| --- | --- |
| FR-CAL-01 | ユーザーは複数の表示単位（月／週／日など）で予定とタスクを確認できる。具体的な表示単位とUIは設計フェーズで確定する。 |
| FR-CAL-02 | ユーザーは前後の期間・今日へ移動できる。 |
| FR-CAL-03 | 日付セルには、その日の予定と、締切日が一致するタスクが表示される。 |
| FR-CAL-04 | 選択中の日付・今日・土曜日・日曜日・祝日が視覚的に区別される。 |
| FR-CAL-05 | 日付を選択すると、その日の予定とタスクの詳細を確認できる。 |
| FR-CAL-06 | 複数日にまたがる予定は、該当する各日に表示される。 |

補足: 祝日判定は日本の祝日を対象とする。表示の詳細（配色・レイアウト・週開始曜日など）は設計フェーズで決める（参考実装の青系/緑系の色分けは必須要件としない）。

### 6.4 予定管理（FR-SCH）

予定のデータ構造は、Google Calendar との同期のため Google Calendar Event に合わせる。登録できる項目は同期要件に従って確定する。

| ID | 要件 |
| --- | --- |
| FR-SCH-01 | ユーザーは予定を追加・編集・削除できる。 |
| FR-SCH-02 | 予定には、予定名（summary）・開始時刻・終了時刻・繰り返し・場所・メモ（description）等を登録できる。項目は Google Calendar Event 互換とする。 |
| FR-SCH-03 | 予定名・開始時刻・終了時刻は必須とする。 |
| FR-SCH-04 | 開始時刻が終了時刻より後の予定は登録できない（バリデーション）。 |
| FR-SCH-05 | 繰り返しは Google Calendar の繰り返し（RRULE / RFC5545）に合わせる。単純な繰り返し（毎日/毎週/毎月/毎年）だけでなく、複雑な RRULE（例: 隔週、毎月第2火曜、平日のみ、N回で終了 等）まで扱えるようにする。 |
| FR-SCH-06 | 繰り返しには開始（DTSTART）と、終了（UNTIL/COUNT）または無期限を指定できる。Google Calendar と整合させる。 |
| FR-SCH-07 | 繰り返し予定の編集・削除では、対象を「この予定のみ／以降すべて／すべて」から選べる（Google Calendar に合わせる。例外インスタンスで表現）。 |
| FR-SCH-08 | 複数日にまたがる予定を登録できる。 |

> 調査メモ（FR-SCH-05/06/07）: Google Calendar の繰り返しは RRULE（RFC5545）で表現され、`DTSTART`＝開始、`RRULE` の `UNTIL`/`COUNT` で終了、いずれも無ければ無期限。「この予定のみ／以降／すべて」は Google Calendar 標準のUXで、API上は例外インスタンスで表現する。本アプリは複雑な RRULE まで対応する方針とし、猶予時間計算（9章）でも各日の発生判定に RRULE 展開を用いる。対応する RRULE の具体的な範囲・展開実装は設計フェーズで詰める。

### 6.5 タスク管理（FR-TASK）

タスクのデータ構造は、Google Tasks との同期のため Google Tasks 互換を基本とする。ただし **所要時間（estimatedTime）は Google Tasks に存在しないため、TiDepo 独自項目**として保持する。

| ID | 要件 |
| --- | --- |
| FR-TASK-01 | ユーザーはタスクを追加・編集・削除できる。 |
| FR-TASK-02 | タスクには、タスク名（title）・締切日（due）・所要時間（estimatedTime, 独自）・メモ（notes）・完了状態（status）を登録できる。 |
| FR-TASK-03 | タスク名・締切日・所要時間は必須とする（葉タスクの場合。親タスクの所要時間は子の合計から算出する）。 |
| FR-TASK-04 | 所要時間は正の数（時間単位）で入力する。 |
| FR-TASK-05 | ユーザーはタスクを完了済みにできる。完了タスクは猶予時間計算の対象から除外する。 |
| FR-TASK-06 | タスクは締切日のカレンダー上に表示される。 |
| FR-TASK-07 | タスクリストでは、タスク名・締切日・所要時間・猶予時間・完了状態を確認できる。 |
| FR-TASK-08 | ユーザーはタスクリストを列（締切日・猶予時間・所要時間など）でソートできる（任意の手動並び替えではなく、列基準のソート）。 |

補足: 完了状態は参考リポジトリには無かった新機能。Google Tasks の status（needsAction / completed）と対応させる。

### 6.6 タスクの入れ子（親子構造）（FR-TREE）

プロジェクトという独立した概念は持たず、タスクの入れ子（親子）ツリーで「まとまり」を表現する。最上位タスクがプロジェクトの役割を兼ねる。

| ID | 要件 |
| --- | --- |
| FR-TREE-01 | ユーザーは大きなタスクを子タスクへ分割できる（入れ子・親子構造、多階層）。 |
| FR-TREE-02 | 最上位タスク（親を持たないタスク）は、配下のタスクをまとめるプロジェクト的な役割を兼ねる。 |
| FR-TREE-03 | 親タスクの所要時間は、常に子タスクの所要時間の合計に一致する（不変条件）。親タスクが本来持つ所要時間は、分割後も保たれる。 |
| FR-TREE-04 | 親タスクを子へ分割した際、明示的に作成した子タスクの所要時間合計が親タスクの所要時間に満たない場合、システムは差分を吸収する子タスク（残余タスク）を自動生成し、合計が親の所要時間に一致するよう保つ。残余タスクの名前は親タスクに由来する（例: 「<親タスク名>（残り）」）。残余タスクはユーザーが特に意識せず扱えるが、猶予時間計算上は葉タスクとして扱う。 |
| FR-TREE-05 | 残余タスクは、子タスクの追加・削除・所要時間変更に応じて自動的に再計算（調整）される。 |
| FR-TREE-06 | 残余タスクは区別して表示するが、所要時間は自動計算のため直接編集できない（他の子タスクを変更すると自動調整される）。 |
| FR-TREE-07 | 子タスクの所要時間の合計が親タスクの所要時間を超えた場合は、親タスクの所要時間を子の合計に合わせて自動的に引き上げる（残余タスクは0となる）。 |
| FR-TREE-08 | 親タスク（特に最上位タスク）で、配下タスクの合計所要時間や進行状況（完了/未完了）を確認できる。 |
| FR-TREE-09 | タスクを別の親へ移動（付け替え）できる。最上位へ出すこともできる。 |

補足: 猶予時間計算は実所要時間を持つ **葉タスク**（自動生成した残余タスクを含む）を対象とする（[9. 猶予時間計算仕様](#9-猶予時間計算仕様)）。Google Tasks への同期も葉タスクのみを対象とする（[6.11](#611-google-tasks-同期fr-gt)）。

> 残余タスクの方式メモ: 親タスクの所要時間を「子の合計」で都度上書きするのではなく、親が本来意図した所要時間を保ったまま、不足分を裏で残余タスク（葉タスク）として保持することで「親＝子の和」を成立させる。これにより、猶予時間計算を葉タスク基準のまま維持できる。残余タスクは区別表示するが編集不可（FR-TREE-06）、子の合計が親を超えた場合は親を自動で引き上げる（FR-TREE-07）。

| FR-GC-05 | Google Calendar 側で作成・更新・削除した予定は、アプリ側へ反映される。 |
| FR-GC-06 | アプリ側と Google Calendar 側の両方で同じ予定を変更した場合の競合解決方針を定める（既定は Google 側優先。詳細は設計フェーズ）。 |
| FR-GC-07 | 同期の結果・状態（最終同期日時、エラー、競合解決等）をユーザーが確認できる。 |

> 調査メモ（FR-GC-02）: Google Calendar API は push 通知（watch channel）と `syncToken` による増分同期の両方に対応するが、本アプリは**ポーリング（`syncToken` 増分）に統一**する。理由: push は通知が「変更あり」のみで結局 `events.list`＋syncToken が必要なうえ、公開 webhook・channel 期限管理・ドメイン検証等の運用が増え、学習対象でもないため。Tasks も push 非対応でポーリングのため、方式が揃う。push（ほぼリアルタイム化）は将来の最適化として残す。`syncToken` 失効（410）時はフル再同期する。自動同期前提のため、初版で固定の同期範囲（過去N日〜未来M日）を必須要件とはしない（範囲指定の要否は設計フェーズで判断する）。
>
> 注（FR-GC-08）: 終日予定・繰り返し例外・参加者/会議リンク等を「アプリ側から完全編集」できるようにするのは難度が高い。MVP の対応範囲（どこまで編集可能にし、どこを表示のみ/Google側編集に委ねるか）は設計フェーズで線引きする。「アプリで編集できない種別」を残さない方針をデータ構造設計の目標とするが、Google 仕様上の制約により完全には排除できない可能性がある（未対応種別が残る場合は警告表示する）。

| ID | 要件 |
| --- | --- |
| FR-GT-01 | ユーザーはアプリのタスクと Google Tasks を双方向に同期できる。 |
| FR-GT-02 | 同期対象は実所要時間を持つ葉タスクのみとする。親タスクは Google Tasks へ同期しない。 |
| FR-GT-03 | 自動同期はポーリング方式で行う（Google Tasks API は push 通知に非対応のため）。`updatedMin` による増分取得、`showDeleted` による削除検知を用いる。 |
| FR-GT-04 | アプリの所要時間（estimatedTime）は Google Tasks に対応項目が無いため同期しない（アプリ内のみ保持）。 |
| FR-GT-05 | Google Tasks 側で作成され、所要時間が不明なタスクは、一時的なタスクとして取り込む（FR-TMP-06）。 |
| FR-GT-06 | 入れ子タスクは葉タスクを Google Tasks へ反映する。親タスクの所要時間は子の合計に一致する前提で扱う。 |

## 9. 猶予時間計算仕様

本アプリの中核ロジック。参考実装（`todolistcalendar2` の `RemainimgTimeUtil.ts`）で検証済みの定義を出発点とするが、**確定仕様ではない**。より適切・正確な計算方法があればそちらを採用する（要調査）。すべての時間は「時間（hour）」単位、日付計算は日単位で行う。

対象は実所要時間を持つ **葉タスク**（自動生成の残余タスクを含む）とし、**完了タスクは除外** する。親タスクの所要時間は子の合計として扱う。

### 9.1 日別予定時間の事前計算（参考方式・要調査）

今日から計算最終日までの各日について、その日に予定が占有する合計時間を求める。

1. 未完了タスクのうち最も遅い締切日と `dateTakeIntoAccount` の **早い方** を計算最終日とする（`dateTakeIntoAccount` 未設定＝無限確定モードの場合は最も遅い締切日を計算最終日とする）。
2. 今日から計算最終日まで1日ずつ進めながら、各日に該当する予定（RRULE 展開・複数日跨ぎを含む）の占有時間を合算する。跨ぎ日はその日の範囲にクリップする。
3. 各日の合計予定時間は **`availableTimePerDay` を上限** とする。

結果は「日付 → その日の予定占有時間（上限あり）」のマップとして保持する。

### 9.2 タスクごとの猶予時間（参考方式・要調査）

対象タスク（未完了の葉タスク）について、以下を順に計算する。

1. **締切までの日数** = 締切日 − 今日（日単位）。
2. **総利用可能時間（予定考慮前）** = 締切までの日数 × `availableTimePerDay`。
3. **予定で使う時間の差し引き**: 今日から「締切」と「`dateTakeIntoAccount`」の早い方（無限確定モードでは締切）までの各日について、9.1 のマップ（上限適用済み）の予定時間を合算して差し引く。
4. **他タスク所要時間の差し引き**: 対象タスクを含む、締切日が対象タスクの締切日 **以前（同日含む）** の未完了葉タスクの所要時間合計を差し引く。
5. **未確定期間の補正**: `dateTakeIntoAccount` が設定され、かつ締切がそれより先にある場合、未確定期間の各日について `(availableTimePerDay − availableTimePerUnscheduledDay)` を差し引く。無限確定モードではこの補正を行わない。

結果が **正** なら余裕あり、**負** なら時間不足を示す。

### 9.3 サボってもいい時間（補助指標）

現在のタスク状況から、1日平均で作業しなくても締切に間に合う余裕時間を算出して表示する（補助的指標であり必須ではない）。算出式は設計フェーズで明文化する。

### 9.4 表示ルール

- 猶予時間が少ないタスク、およびそれより締切が早いタスクは強調表示する（FR-REM-07）。
- 猶予時間が負のタスクは時間不足として明示する（FR-REM-06）。

> 注: 本章のロジックは参考実装に基づく出発点であり、確定ではない。計算方法は要調査とし、より良い方式を採用しうる。いずれの方式でも、猶予時間計算・繰り返し予定の発生判定・日付処理は純粋ロジックとして実装し単体テストで保護する（[10.4](#104-品質テスト方針)）。

- バックエンド: **オニオンアーキテクチャ**（ドメイン中心、依存方向を内向きに保つ）。猶予時間計算等のコアロジックはドメイン層の純粋ロジックとして配置する。
- フロントエンド: **FSD（Feature-Sliced Design）** を基本とする。
- **Google 連携（同期）は腐敗防止層（ACL）／独立モジュールとして隔離する。** 双方向同期の複雑さ（競合解決・冪等性・トークン更新・ポーリング/syncToken 管理等）をドメインへ漏らさず、ドメイン層は外部サービス（Google）に依存しない状態を保つ。
- 優先軸は「学習の深さ」。学習目的のため難所も基本的に自前実装するが、過剰抽象（単純CRUDへのリッチドメイン強制）は避け、ドメインの作り込みは猶予時間計算とタスク木に集中させる。
- いずれも本アプリに不適切と判断した場合は、より適切なアーキテクチャを採用しうる。

### 10.3 認証・データ永続化

- ログインは Google ＋ メール/パスワードに対応し、OAuth2.0 / OIDC（認可コードフロー＋PKCE）を用いる。
- **BFF パターンを採用する。** Spring を OAuth2 クライアントとし、ブラウザ（React SPA）↔サーバ間は httpOnly セッションCookie で連携、OAuth トークンはサーバ側で保持する（ブラウザにトークンを置かない＝XSS によるトークン奪取を避ける）。PKCE はサーバ↔外部IdP/Google の認可コードフローで用いる。メール/パスワードログインも同じサーバセッションに帰着させる。
- データはユーザー単位で分離・永続化し、再ログイン後も保持する。
- ロール（ADMIN / USER）に基づき操作・画面を出し分ける。
- Google 連携（Calendar / Tasks）は、ログインとは別に該当スコープの認可を取得する。トークンはサーバ側で安全に保管する。

### 10.4 品質・テスト方針

- 猶予時間計算・繰り返し予定の発生判定・日付処理は純粋ロジックとして実装し、単体テストで保護する（移植・方式変更時の回帰防止）。
- CI で lint・型チェック・テスト・ビルドを自動実行する。
- 参考実装の既知の挙動を、計算方式の比較・回帰判断の基準として活用する。

### 10.5 非機能要件（NFR）

| ID | 要件 |
| --- | --- |
| NFR-01（監査） | 主要エンティティの変更履歴（誰がいつ何を変更したか）を保持する。基本監査項目（createdAt/updatedAt/createdBy 等）に加え、変更履歴を残す（Hibernate Envers 等を想定）。 |
| NFR-02（レスポンシブ） | 単一の React Web アプリをレスポンシブ化し、PC／スマートフォン両方で主要操作を行える。 |
| NFR-03（セキュリティ） | 認証/認可の適切な実装、機密情報（OAuth クライアントシークレット、トークン、AWS 鍵等）の安全な管理、入力検証を行う。 |
| NFR-04（データ分離） | ユーザー単位でデータを分離し、他ユーザーのデータへアクセスできないようにする。 |

## 11. 制約・前提

- MVP はマルチユーザー（データはユーザー単位で分離）を前提とするが、複数ユーザー間での実データ共有・コラボレーションは対象外（将来構想）。ロールは主に管理者機能の出し分けで意味を持つ。
- Google 同期は、最初は専用のテストカレンダー／テストアカウントで試すことを推奨する。
- Google Tasks は push 通知に非対応のため、タスクの自動同期はポーリングとなり即時反映はできない。所要時間は Google 側に保持できない。
- Google Calendar の一部の種別（複雑な繰り返し・終日・参加者付き・特殊予定）は、アプリ側からの完全編集が難しい場合がある。アプリ未対応の種別が残る場合は警告表示する。
- 個人開発・趣味の範囲のため、可用性・SLA 等の厳密な保証は本リリースの対象外とする。

## 12. 将来構想

本リリースのスコープ外。データモデル・アーキテクチャはこれらの拡張を阻害しない構造を目指す。

### 12.1 複数ユーザー間の共有・チーム/組織管理

- Slack など、Google 以外のサービスとの連携。価値を見出せたら検討する。

### 12.4 PWA / ネイティブアプリ

- レスポンシブWebに加え、オフライン対応の PWA やネイティブアプリ化を検討する。

## 付録B: 推奨実装順（段階リリースの目安）

今回の MVP は範囲が広いため、段階的に構築することを推奨する（順序・粒度は設計フェーズで調整）。

1. **基盤**: プロジェクト雛形（Docker / CI/CD）、RDB（PostgreSQL）、認証（メール/パスワード＋Google、OAuth2.0/PKCE）、ロール、ユーザー単位データ分離、監査基盤。
2. **コア**: 予定・タスク（完了含む）・一時タスク・設定・猶予時間計算（葉タスク・完了除外・無限確定モード）・カレンダー表示。
3. **タスク入れ子**: タスクの親子分割、残余タスク、集計・進行状況。
4. **Google Calendar 同期**: まず手動の双方向同期で安定させ、ポーリング（syncToken 増分）による自動同期へ拡張。
5. **Google Tasks 同期**: 葉タスクの双方向同期（ポーリング、Google 由来は一時タスク取り込み）。
6. **仕上げ**: 管理者画面、変更履歴（監査）の充実、レスポンシブ最終調整。
